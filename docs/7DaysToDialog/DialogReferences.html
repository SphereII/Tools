Dialog References

Dialogs.xml is XML that handles the dialog options for a NPC in 7 Days To Die.


Each dialog entry has its own dialog ID, which defines a unique identifier which NPC can use to reference it, along with a starting statement. The dialogs.xml's dialog ID is a reference to the npc.xml's dialog_id attribute. The npc_info's id attribute refers to the NPCID in the entityclasses.xml.


dialogs.xml

<dialogs>

       <dialog id="trader" startstatementid="start">

             <!-- snip -->

       </dialog>

</dialogs>



npc.xml

<npc>

       <npc_info

               id="traitorjoel"

               name="Traitor Joel"

               name_key="npc_traitorJoel"

               faction="whiteriver"

               portrait="npc_joel"

               greeting_type="nearby"

               stance="Like" voice_set="trader"

               trader_id="1" dialog_id="trader" />

             <!-- snip -->

</npv>



entityclasses.xml

<entity_class name="npcTraderJoel" extends="npcTraderTemplate">

       <property name="Tags" value="entity,npc,human"/>

       <property name="Mesh" value="#Entities/Traders?JoelRagdoll.prefab"/>

       <property name="NPCID" value="traitorjoel"/>

</entity_class>



A typical dialog entry will contain one or more statements with one or more responses.  A statement is what the NPC can say to you, and a response is what you can say to the NPC.


A statement node contains one or more response_entry. These entries point to a response node that has the same id as the response_entry's id.


This response_entry is only a reference to the actual response; this allows the same response to be re-used by different statements.


A response entry can contain requirements that determine if the response should show on the player's screen, what text to display as part of the response, and any special action that should happen when a response is clicked.


<statement id="start" text="dialog_trader_statement_start">

       <response_entry id="shop" />

       <response_entry id="jobsnone" />

       <response_entry id="jobshave" />

       <!-- <response_entry id="background" /> -->

       <response_entry id="admin" />

       <response_entry id="done" />

</statement>


<!-- Responses -->

<response id="shop" text="dialog_trader_response_shop">

       <action type="Trader" id="trade" />

</response>

       

<response id="jobsnone" text="dialog_trader_response_jobsnone" nextstatementid="nojobs" >

       <requirement type="QuestStatus" value="InProgress" requirementtype="Hide" />

</response>


<response id="jobshave" text="dialog_trader_response_jobshave" nextstatementid="currentjobs" >

       <requirement type="QuestStatus" value="NotStarted" requirementtype="Hide" />

       <action type="AddJournalEntry" id="questTip" />

</response>


Statement:


A statement is what the NPC will say to you, setting up a conversation chain. It has a unique ID that is used as a reference by other parts of the dialog system, and a text attribute. This text attribute can either be a Localization.txt key entry, or it can contain the plain text you want to display.


A statement can contain one or more response_entry to provide more dialog options for the player to say to the NPC, furthering the conversation. However, a statement does not require a response_key. If the statement has a nextstatementid attribute, and there are no response_entry, then an auto-generated "[ NEXT ]" is added by the game. Clicking on this will jump to the statement defined by the nextstatementid.


Response:


A Response is what the player can say back to the NPC. It contains an ID reference that is used by the response_key from a statement dialog, a text attribute that can be a Localization.txt key entry or it can contain the plain text you want to display.


A response node can contain other XML nodes.  A requirement node inside a response node can affect the visibility of the response. That is, if the conditions of the requirement(s) is true, then the response option will show up for the player to select. If there are no requirements inside the response, then the response is shown.


One or more Action nodes can also exists in the Response. The action nodes can perform an action that can affect the player or NPC when selected. This could range from giving the player a quest, a buff, get an item from an NPC or give an item from an NPC.


A Response ID must be unique; however, the text value of a response can be identical to another one. This allows the player to give the same answer to the NPC, but have different actions depending on requirements. For example, you could ask the NPC about a particular trader.  The NPC may have different statements to say to you depending on if you have talked to the particular trader or not. A requirement could be created where one response shows up if you've met the trader before, while another response shows up if you have never met the trader before. The response text would be the same, but it would have two different results.


Example of responses that share the same text.

<response id="response_1374956358" text="What can you tell me about Trader Joel?" nextstatementid="statement_1825662454">

      <requirement type="HasBuffSDX, Mods" value="buffMetTraderJoel" requirementtype="Hide" Hash="Requirement_1335115275" />

</response>

<response id="response_1374956358" text="What can you tell me about Trader Joel?" nextstatementid="statement_99999999">

      <requirement type="!HasBuffSDX, Mods" value="buffMetTraderJoel" requirementtype="Hide" Hash="Requirement_1335115275" />

</response>



The above example shows two different responses with "What can you tell me about Trader Joel?". Only one will show up at a time since we have a requirement set on each. The first option will only display when the player has the buff BuffMetTraderJoel. If the player does not have that buff, the other "What can you tell me about Trader Joel?" shows up. The nextstatementID points to different statement IDs, giving you a different conversation chain.
